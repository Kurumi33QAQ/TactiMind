# TactiMind 长期开发 Checklist

> 说明：这份清单用于约束后续开发方向。每完成一项，就把对应复选框改成 `[x]`，避免项目在迭代过程中偏离“足球战术分析 AI Agent 系统”的主线。

## 一、已经完成的基础能力

- [x] 完成 Spring Boot 后端骨架、WebSocket 推送和本地 JSON 比赛事件流
- [x] 完成比赛状态聚合，支持比分、射门、射正、角球、危险进攻、控球率等统计
- [x] 接入 Redis 实时状态缓存和 MySQL 持久化能力
- [x] 搭建 Python FastAPI Agent 服务
- [x] 实现 `DataAgent -> TacticsAgent -> VerifyAgent` 的规则型分析链路
- [x] 实现 `evidence / confidence / riskLevel` 防幻觉输出结构
- [x] 完成比赛目录、比赛搜索、历史任务、Agent Trace 和结构化报告接口
- [x] 拆分独立 Vue 前端模块，完成中文化比赛库和实时战术工作台
- [x] 优化前端布局、图表、历史任务页、队伍区分和分析卡片分层

## 二、接下来优先推进的产品体验

- [x] 进入比赛后先展示比赛详情，不自动开始分析
- [x] 将比赛模拟从“按事件跳出”升级为“按比赛时钟从 0 分钟平滑推进”
- [x] 增加演练倍速控制，支持 `0.5x / 1x / 2x / 4x / 8x`
- [x] 增加时间轴跳转能力，支持从指定分钟继续演练
- [x] 继续优化事件流、分析卡片、时间轴和图表之间的联动体验

## 三、补强分析依据

- [ ] 接入比赛上下文数据：阵型、首发阵容、替补席、球员位置、球员状态、关键球员
- [ ] 增加球员能力标签，例如速度、传球、对抗、压迫、终结、持球推进
- [ ] 补充双方主教练信息，并在 Agent 分析中作为战术风格依据
- [ ] 用站位图展示完整首发 11 人，替补球员排列在站位图下方
- [x] 在比赛详情页展示战术资料 Profile，并明确模拟资料的数据说明
- [x] 将战术资料 Profile 作为 Agent 请求上下文传入 Python Agent
- [x] 让 Agent 从“只看事件流”升级为“事件流 + 阵型 + 阵容 + 球员状态”联合分析
- [x] 扩展事件字段，例如进攻方向、区域、攻防转换、关键传球、抢断、丢失球权

## 四、建设知识库与 RAG

- [ ] 建立本地战术知识库，沉淀高位逼抢、低位防守、边路压制、肋部进攻、弱侧转移、攻防转换等知识
- [ ] 新增 `RetrievalAgent`，负责根据比赛现象检索战术知识
- [ ] 先实现轻量版 RAG：先检索知识，再生成战术结论
- [ ] 在知识规模扩大后，再按需要接入向量数据库做语义检索
- [ ] 持续保证每条结论都能关联比赛证据和知识来源

## 五、强化 Multi-Agent 工程能力

- [ ] 明确 `DataAgent / RetrievalAgent / TacticsAgent / VerifyAgent / ReportAgent` 的输入、输出和工具边界
- [ ] 把工具能力标准化，例如 `load_match_events`、`calculate_match_stats`、`retrieve_tactical_knowledge`、`verify_report_evidence`
- [ ] 将赛后结构化报告也接入真正的 Agent 流程，减少 Java 侧固定模板生成
- [ ] 接入 LLM 后保留 VerifyAgent，继续执行证据校验和低置信度降级
- [ ] 在接入 LLM 后增加 SSE 流式分析输出，用于展示逐步生成报告的过程

## 六、状态恢复与存储演进

- [ ] 完善 Redis 比赛状态快照和恢复能力
- [ ] 支持根据事件日志恢复指定比赛进度
- [ ] 评估复杂 Agent 上下文、多版本草稿和嵌套快照的真实需求
- [ ] 仅在确有必要时再引入 MongoDB，不为了堆技术提前接入

## 七、体现“不是直接调用 LLM API”

- [ ] 持续强化工具调用、Agent Trace、RAG 检索、证据链、置信度和防幻觉校验的可见展示
- [x] 在前端明确展示“哪个 Agent 调用了什么工具、基于哪些数据得出什么结论”
- [ ] 增加比分预测链路的可解释展示，用“输入数据、预测依据、置信度、赛后误差评估”体现系统不是简单问答
- [ ] 在 README 中增加“与直接调用 LLM API 的区别”章节
- [ ] 确保最终演示能体现：系统不是单次问答，而是完整的 Agent 工作流

## 八、比分预测与准确度评估

> 接入时机：建议在“比赛上下文数据、球员能力、轻量 RAG、Agent Trace”稳定后加入。历史比赛可以先做赛中滚动预测，未开赛比赛再做赛前预测，避免早期预测只能靠规则硬猜。

- [ ] 设计 `PredictionAgent` 职责边界，明确输入为比赛状态、最近事件、阵容球员、战术知识和历史基础数据
- [ ] 为历史比赛增加赛中动态预测：随着比赛时钟推进，输出预测最终比分、置信度、预测依据和风险说明
- [ ] 在前端将“最终比分预测”与“Agent 战术分析”标题放在同一区域，展示当前预测比分和变化趋势
- [ ] 为未开始比赛增加赛前预测：基于球队、阵容、球员状态、基础历史数据和战术知识生成预测
- [ ] 增加预测结果持久化，记录预测时间点、输入摘要、预测比分、置信度、证据和模型版本
- [ ] 比赛结束后计算预测误差，展示“预测准确度 / 命中比分 / 命中胜平负 / 偏差分钟趋势”
- [ ] 接入 `VerifyAgent` 校验预测依据，数据不足时必须提示“当前数据不足，预测仅供演练参考”
- [ ] 在 README 和面试总结中说明：比分预测模块通过工具调用和赛后评估区别于普通 LLM API 调用
